# Herbound 产品功能审查文档

审查日期：2026-06-18  
审查范围：`hermes-agent-main` 当前项目、Herbound 前端 UI、FastAPI 后端、DeepSeen SDK 插件、跨境电商 skill、生产部署配置。

## 1. 产品定位

Herbound 当前已经不是通用 Hermes Agent 的简单换皮版本，而是以跨境电商业务为核心的企业内部智能体。它的核心目标是：

- 先理解用户的跨境电商业务需求。
- 优先选择公司提供的 DeepSeen SDK 工具集处理任务。
- 对 SDK 输出做用户友好的结构化展示，不进行业务含义上的二次润色。
- 支持图片、视频、商品链接、本地文件等资源输入。
- 将生成结果、分析结果、图片、视频等内容直接在 Web 对话框中展示。
- 使用 FastAPI 作为唯一业务后端，前端沿用 hermes-web-ui 的 UI 壳。

当前项目已经具备可运行的产品雏形，但距离生产环境还需要完成权限、安全、会话一致性、资源上传、进度语义、业务页面收敛等稳定化工作。

## 3. 当前主要业务流程

### 3.1 标准对话流程

1. 用户登录 Herbound。
2. 用户在设置页填写 DeepSeen key。
3. 用户进入聊天页。
4. 用户输入跨境电商需求，例如竞品分析、图片二创、视频生成、达人分析等。
5. 智能体识别意图。
6. 如果参数不足，智能体用自然语言追问必要信息。
7. 如果参数足够，智能体选择对应 DeepSeen 工具。
8. 工具执行时向前端推送进度。
9. 前端显示生成进度入口和详情。
10. 工具返回结果后，智能体将结果转为用户友好的结构化内容。
11. 聊天框展示文字、图片、视频、链接等结果。
12. 会话进入历史记录。

### 3.2 媒体生成流程

1. 用户上传图片、视频或提供商品链接。
2. 前端将资源上传到后端。
3. 后端返回可访问资源地址或文件 ID。
4. 对话提交时资源信息随用户消息传入。
5. 智能体调用图片或视频生成工具。
6. DeepSeen SDK 返回任务状态、进度和最终媒体资源。
7. 前端展示生成进度和最终结果。

## 4. 存在的功能缺陷与风险

### P0：上线前必须修复

#### 4.1 跨境 skill 文件存在中文乱码风险

文件：

- `skills/crossborder-deepseen/SKILL.md`

审查中发现该文件部分中文内容出现乱码。这个文件直接影响智能体的业务路由、参数追问、结果展示规则。若 prompt 中存在乱码，模型可能无法稳定理解业务字段，也可能向用户输出异常文本。

建议：

- 统一用 UTF-8 重写该 skill。
- 将所有用户可见字段改成中文业务表达。
- 增加一份工具字段含义映射表，避免模型直接使用 SDK key 名。

#### 4.2 生产配置中仍存在本机全局路径

文件：

- `.hermes/config.yaml`

当前 MCP 配置中存在类似 `C:\Users\Administrator\.hermes-web-ui` 的本机全局路径。这与“所有配置都在项目文件夹中、上线服务器不能依赖本地全局资源”的生产要求冲突。

建议：

- 移除所有用户目录绝对路径。
- 将运行目录改为项目内或容器内路径，例如 `/opt/app`、`/opt/data`。
- 为开发环境和生产环境拆分配置模板。

#### 4.3 Web 登录安全仍不满足生产标准

当前已有 JWT 登录，但还存在生产风险：

- 默认账号密码过弱，当前产品仍能看到 `admin / 123456` 这样的初始化逻辑。
- JWT 有效期较长，缺少服务端会话撤销机制。
- 修改密码后旧 token 是否失效不明确。
- JWT 存在 localStorage 中，存在 XSS 后 token 被读取的风险。
- 注册接口和 public allowlist 的意图不完全一致，容易造成“接口存在但不可用”或“错误开放注册”的安全边界问题。
- 多用户、多 profile 权限边界还需要逐接口确认。

建议：

- 首次启动强制修改默认密码。
- 生产环境默认关闭公开注册。
- 增加 token version 或 session 表，实现改密后旧 token 失效。
- 明确 super admin、admin、普通用户权限模型。
- 对所有 profile、插件、文件、日志、DeepSeen key 接口做权限审计。

#### 4.4 DeepSeen 进度语义不可靠

当前 UI 会展示 SDK 上报的进度，但实际出现过“进度 100%，状态仍为 generating”的情况。

这说明当前百分比不等于任务完成，只能代表 SDK 或上游任务的某个阶段进度。若直接展示 100%，用户会认为任务已经完成，体验会产生冲突。

建议：

- 前端展示逻辑以状态为准，`completed` 才显示完成。
- 当状态不是终态时，即使进度为 100%，也展示为“处理中，等待结果确认”。
- 进度文案中明确区分“处理进度”和“最终结果状态”。
- 后端统一转换 SDK 进度事件，避免不同工具进度语义不一致。

#### 4.5 长任务超时时间与需求不一致

用户明确要求生成视频超时时间设置为 1 小时，但当前插件中默认超时时间为：

- `_DEFAULT_TIMEOUT_MS = 7200000`
- 视频工具描述中也出现 “2-hour wait timeout”

这代表当前实现是 2 小时，与产品需求不一致。

建议：

- 明确最终产品策略：图片、视频、分析任务分别允许等待多久。
- 若需求确认为 1 小时，将视频超时统一改为 `3600000`。
- 前端在进度弹窗中显示预计最长等待时间。

#### 4.6 用户 DeepSeen key 的传递链路需要彻底验证

当前 DeepSeen key 存在用户级存储，但工具执行时依赖当前请求上下文和环境变量式传递。

风险：

- 如果后台任务、WebSocket、子进程或 runner 没有正确携带用户上下文，可能读取不到用户 key。
- 如果 fallback 到 `local` key，可能导致不同用户之间的 key 隔离不清晰。

建议：

- 工具调用必须显式绑定 user_id。
- 子进程调用 DeepSeen SDK 时显式传递当前用户的 key。
- 禁止生产环境 fallback 到不明确的 local key。
- 增加“当前使用的是哪个用户 key”的后端审计日志，但不要在前端泄露完整 key。

### P1：影响核心体验，需要优先修复

#### 4.7 会话拆分与历史记录一致性仍有风险

用户已经反馈同一窗口对话会被拆到不同会话。当前前端和后端之间同时存在 UI session、native Hermes session、history session 等概念，且原生 session ID 有格式限制。

风险：

- 刷新页面后找不到原会话。
- 同一聊天窗口提交多轮后产生多个后端会话。
- 历史列表和聊天详情不是同一数据源。

建议：

- 统一一个会话 ID 作为产品主键。
- 前端路由、WebSocket、历史记录、删除会话全部使用同一 ID。
- 删除“临时 UI 会话”和“真实后端会话”的双轨逻辑，或建立明确映射表。
- 增加端到端测试：同一会话连续发送 3 条消息、刷新、切换页面、恢复后仍在同一会话。

#### 4.8 Koa 功能迁移尚未形成完整验收清单

产品方向是只保留 FastAPI 一个后端。当前登录、基础认证、聊天、插件等核心功能已迁移或对接，但 hermes-web-ui 原本的 Koa 后端里可能还有设备、文件、编码智能体、TTS/STT、部分用户设置等功能没有完全迁移。

建议：

- 建立“原 Koa 接口 -> FastAPI 接口”的对照表。
- 每个前端页面标记：已接入、半接入、占位、隐藏。
- 未接入的页面不要在生产导航中直接暴露。

#### 4.9 文件上传到 OSS 的生产方案还未最终落地

当前本地上传和资源 URL 注入已经能跑，但生产环境更适合使用 OSS/CDN。尤其图片二创、视频二创需要 SDK 能访问到稳定的公网资源。

建议：

- 服务端上传后立即转存 OSS。
- 聊天消息中只传 OSS URL 和资源元数据给智能体。
- 本地文件 ID 仅作为审计和下载引用，不直接给 SDK。
- OSS URL 要有生命周期、安全策略和访问权限设计。

#### 4.10 工具结果展示仍有泄露 raw 字段的可能

当前已经做了用户友好展示，但历史中仍出现过 JSON key、内部字段名、原始结果结构暴露给用户的情况。

建议：

- 为每个 DeepSeen 工具建立固定展示 schema。
- 只展示文档规定的字段。
- 未识别字段默认折叠到“调试信息”，生产环境不展示。
- 所有结果展示走统一 renderer，不让模型自由拼接 raw JSON。

#### 4.11 技能页和插件页仍偏通用 Hermes

用户已经指出 `/hermes/skills` 中很多内容与跨境电商业务无关。当前虽然可以改成跨境业务展示，但它和真实 tool registry / skill registry 的同步关系还不够清楚。

建议：

- 技能页只展示 Herbound 业务技能。
- 每个技能绑定真实 DeepSeen 工具。
- 展示输入要求、输出结果、适用场景、最近调用状态。
- 通用 Hermes 技能从生产导航中隐藏或移动到管理员高级页面。

### P2：中期优化

#### 4.12 插件管理页需要区分“产品插件”和“系统插件”

当前插件页能看到 Hermes 原生插件和 Herbound 业务插件。对普通业务用户来说，只有 DeepSeen 相关能力才是可理解的。

建议：

- 普通用户只看到业务能力。
- 管理员才能看到系统插件。
- 插件启停、升级、删除需要权限控制和操作审计。

#### 4.13 导航信息量过大

当前保留了 Hermes Web UI 的完整导航，包括 MCP、终端、设备、编码智能体、看板、性能、日志等。对跨境电商业务用户来说，这些不是日常主流程。

建议：

- 普通用户导航只保留：聊天、历史、素材/文件、任务、技能、设置。
- 管理员导航保留：模型、插件、日志、用量、性能、用户、系统配置。
- 高危能力如终端、文件系统、MCP 管理默认只给超级管理员。

#### 4.14 品牌命名仍需统一

项目中仍存在 Hermes、Hermes Web UI、dashboard、TUI 等历史命名。产品层应统一为 Herbound。

建议：

- 用户可见文案统一为 Herbound。
- 开发者文档可以说明 Herbound 基于 Hermes Agent 改造。
- 页面标题、默认头像、设置页、错误提示、登录页全部统一品牌。

#### 4.15 结果翻译策略需要固定边界

当前要求根据用户语言翻译展示内容，但不改变原文语义。这个策略正确，但需要防止模型自由改写工具结果。

建议：

- 工具结果先结构化。
- 字段标题本地化。
- 字段值只做直译，不做总结、发挥或营销润色。
- 对金额、地区、平台、时间、URL 不翻译。

#### 4.16 运维可观测性不足

DeepSeen 调用失败时，用户只能看到失败文案，开发侧需要更完整的链路追踪。

建议：

- 每次工具调用生成 request_id。
- 记录 user_id、session_id、tool_name、job_id、耗时、状态、错误码。
- 前端错误详情给用户友好信息，管理员日志保留完整技术信息。
- 对 SDK API 错误、网络错误、参数错误、超时错误分别分类。

### P3：体验与工程优化

#### 4.17 移动端和小屏体验未完整审查

当前聊天、进度弹窗、富媒体结果在桌面端基本可用，但移动端还需要测试。

建议：

- 验证聊天框上传多文件。
- 验证视频预览。
- 验证长文本报告。
- 验证进度浮窗不遮挡输入区。

#### 4.18 自动化测试覆盖不足

当前产品改造涉及前端、FastAPI、WebSocket、插件、Node SDK runner，链路较长。

建议增加测试：

- 登录成功和失败。
- 保存 DeepSeen key。
- 上传文件并在消息中带资源。
- 同一会话多轮对话不拆分。
- DeepSeen 工具成功、失败、超时三类结果。
- 图片和视频结果富媒体渲染。
- 删除会话。

## 7. 各 DeepSeen 工具的产品展示要求

### 图片智创 / 图片二创

应展示：

- 任务状态。
- 生成图片。
- 图片下载链接。
- 输入图片或商品链接。
- 主要提示词或用户需求。
- 失败原因。

不应直接展示：

- 原始 JSON。
- 内部字段名。
- SDK 调试字段。

### 视频智创 / 视频二创

应展示：

- 任务状态。
- 生成视频。
- 视频下载链接。
- 封面图。
- 输入素材。
- 生成说明。
- 失败原因。

需要额外展示：

- 长任务进度。
- 最长等待时间。
- 当前是否仍在等待最终结果。

### 单竞品分析

应展示：

- 商品基础信息。
- 价格与市场信息。
- 卖点分析。
- 用户评价或反馈。
- 竞争机会。
- 优化建议。
- 商品来源和素材来源。

### 多竞品分析

应展示：

- 分析关键词。
- 目标市场。
- 代表产品列表。
- 价格带。
- 共性卖点。
- 差异化机会。
- 风险点。
- 建议切入方向。

### 产品报告

应展示：

- 产品概览。
- 市场机会。
- 目标用户。
- 定价建议。
- 内容素材建议。
- 平台策略。
- 风险与注意事项。
- 行动建议。

### 达人分析 / 达人评分

应展示：

- 达人基础信息。
- 匹配产品。
- 目标市场。
- 粉丝画像。
- 内容风格。
- 带货能力。
- 合作风险。
- 推荐合作方式。
- 综合评分。

用户追问参数时，不应展示 `product_name`、`target_market`、`sample_tier` 这类字段名，应改成：

- 产品名称。
- 目标市场。
- 目标价格。
- 商品类目。
- 竞品名称。
- 目标用户年龄。
- 目标用户性别。
- 分析深度。

### 视频分析

应展示：

- 视频基础信息。
- 内容结构。
- 卖点表达。
- 节奏与脚本。
- 画面元素。
- 适合平台。
- 可复用创意点。
- 优化建议。

## 8. 上线验收清单

### 核心链路

- 用户能登录。
- 用户能保存 DeepSeen key。
- 用户能上传图片和视频。
- 用户能在同一会话连续多轮对话。
- 会话刷新后不丢失。
- 会话删除可用。
- 图片二创能返回图片。
- 视频二创能展示长任务进度。
- 单竞品、多竞品、产品报告、达人分析均按文档展示。

### 安全链路

- 默认密码不可直接用于生产。
- 未登录访问业务接口返回 401。
- 普通用户不能访问管理员接口。
- 不同用户 DeepSeen key 隔离。
- 日志不打印完整 key。
- 上传文件有大小和类型限制。

### 生产链路

- Docker 启动成功。
- 静态前端可访问。
- FastAPI 健康检查通过。
- Nginx 代理正确。
- DeepSeen SDK runner 可执行。
- OSS/CDN 资源可被 SDK 访问。
- 长任务不中断。
- 日志可定位失败原因。

## 9. 结论

Herbound 当前已经具备跨境电商智能体的核心骨架：Web UI、FastAPI 后端、DeepSeen SDK 工具、用户 key、上传、富媒体结果、生成进度、历史会话等关键模块都已经有实现基础。

当前最大问题不是“有没有功能”，而是“生产一致性和业务收敛还不够”。尤其是 skill 乱码、本机路径、登录安全、会话拆分、进度语义、DeepSeen key 隔离、OSS 资源链路和结果展示 schema，需要作为上线前最高优先级处理。

建议下一步先完成 P0 和 P1 缺陷修复，再进行页面裁剪和业务化重构。这样 Herbound 才能从一个改造中的 Hermes Web UI，稳定变成公司专用的跨境电商智能体产品。
